## Pond usage → PondDevices data flow (step‑by‑step)

This document explains how Pond usage data is ingested from SFTP, staged, transformed, and synced into device usage and Pond device records. It also lists the stored procedures (SPs) used at each step.

### High-level flow
- Trigger (SQS or CloudWatch) → Orchestrate per Provider
- Download and parse Pond usage file(s) → Bulk insert to staging
- Update canonical usage from staging (SP)
- Sync devices from Pond usage (SP)

### Entry points and orchestration
1) Function handler
- `AltaworxPondGetDeviceUsage.Function.FunctionHandler` delegates to `ProcessEventAsync`.
  - With SQS event: iterates records and calls `ProcessEventRecordAsync`.
  - Without SQS event: calls `StartDailyDeviceUsageProcessingAsync` with `isFromCloudwatchEvent = true` (hourly/daily kick-off).

2) Per-provider queueing
- `StartDailyDeviceUsageProcessingAsync`
  - If invoked by CloudWatch (hourly/daily), first clears staging via SP: `POND_TRUNCATE_USAGE_STAGING`.
  - Enumerates service providers for Pond using `ServiceProviderCommon.GetNextServiceProviderId` which calls SP `usp_DeviceSync_Get_NextServiceProviderIdByIntegration`.
  - Enqueues SQS messages to process each provider (device usage download + subsequent steps).

3) Message routing per provider
- `ProcessEventRecordAsync` routes based on message flags:
  - If `InitializeProcessing == true`: re-run `StartDailyDeviceUsageProcessingAsync`.
  - If `PondSyncDataStep > 0`: go to `ProcessPondUsageDataSync` (post-download DB steps).
  - Else: go to `ProcessDailyUsage` (download + stage usage).

### Download and stage Pond usage
4) Load SFTP credentials
- `context.SettingsRepo.GetPondDeviceSettings(serviceProviderId, ...)` provides `PondSFTPUsername/Password/Server/Path`.

5) Determine files to download
- Computes date window from `DaysToKeep` and `CheckFilesMissedThresholdDays`.
- Fetches already-downloaded files from DB via SP: `POND_GET_FILES_DOWNLOADED` (date range).
- Lists SFTP directory, filters names by `RegexConstants.REGEX_MATCH_POND_DEVICE_USAGE_SFTP_FILE` and `.csv`, then selects newest within window, capped by `LimitNumberOfFilesPerInstance`.

6) File-level bookkeeping (before reading)
- Lookup existing file id via SP: `POND_GET_SFTP_FILENAME_ID`.
- Lookup saved record count via SP: `POND_GET_RECORD_COUNT_FROM_DATABASE` (supports batched/partial processing).

7) Download and parse
- Downloads the CSV over SFTP into memory.
- Parses to a `DataTable` using `PondUsageReportReader.ReadBatchedRecords(...)` (supports classic and new CDR formats):
  - Columns include: `ServiceProviderId, FtpFileId, FileRowNumber, CDRID, ICCID, Type, ConnectTime, CloseTime, Duration, Direction, CalledParty, CallingParty, CountryISO3, CountryName, MCC, MNC, IMSIId, IMSINumber, DateCreated`.
  - Applies `UsageRowsCountLimit` so large files are processed in batches; if limit reached, the file is re-queued to continue.

8) Persist file mapping and stage usage
- If not previously known, inserts a file mapping via SP: `POND_INSERT_USAGE_TABLE_NAME_MAPPING` (records file path and times) and captures `FtpFileId`.
- Updates staged rows with `FtpFileId`.
- Bulk-inserts the usage rows to staging table via SQL bulk copy:
  - Destination table: `DatabaseTableNames.POND_DEVICE_USAGE_STAGING`.
  - Column mappings auto-generated by `SQLBulkCopyHelper.AutoMapColumns`.

9) Handle empty/blank files and retries
- If no usage records read:
  - If file unknown: insert as blank via `POND_INSERT_USAGE_TABLE_NAME_MAPPING` with `@IsBlank = true`.
  - Else: mark existing file id as blank via `POND_UPDATE_USAGE_FILE_AS_BLANK`.
- On success for retry cases: update status via `POND_UPDATE_FILENAMES_DOWNLOAD_STATUS`.
- Failed downloads are written to `DatabaseTableNames.POND_SFTP_FILE_DOWNLOAD_STATUS` via bulk copy and re-queued (`ProcessDownloadFileAgain`).
- If not end-of-file (partial batch), the file is re-queued to continue (`SendMessageToQueueDownloadAsync`).

10) Advance to DB sync steps
- After all files for the provider are handled (or none needed), enqueue a message with `PondSyncDataStep = UpdatePondUsageFromStaging`.

### Database sync steps (post-staging)
11) Step 1 – Merge staging into canonical Pond usage
- `ProcessPondUsageDataSync` calls `UpdatePondUsageFromStaging`, which executes SP:
  - `POND_UPDATE_DEVICE_USAGE_FROM_STAGING(@ServiceProviderId)`
- Purpose: validate and merge staged rows into the canonical Pond usage table(s), deduplicating by file/row, and preparing normalized usage records.

12) Step 2 – Sync devices from Pond usage
- Then `ProcessPondUsageDataSync` enqueues and runs `UpdateDeviceUsageFromPond`, which executes SP:
  - `POND_DEVICE_SYNC(@ServiceProviderId)`
- Purpose: propagate canonical Pond usage into device-level aggregates and/or device tables ("PondDevices"), updating counts, statuses, and related device usage tables.
- If triggered from a non-CloudWatch SQS flow, a cleanup message is sent to the Jasper device cleanup queue when this step completes.

13) Optional FTP cleanup
- `CleanUpFtp` deletes older usage files on SFTP based on `DaysToKeep` retention.

### Stored procedures used (by category)
- Orchestration
  - `usp_DeviceSync_Get_NextServiceProviderIdByIntegration` – get next provider id per integration
- Staging lifecycle
  - `POND_TRUNCATE_USAGE_STAGING` – clear staging at job start (hourly/daily run)
- File tracking and retries
  - `POND_GET_FILES_DOWNLOADED` – list already-downloaded file paths (date range)
  - `POND_GET_SFTP_FILE_DOWNLOAD_STATUS_ID` – get failed download ids
  - `POND_GET_SFTP_FILE_DOWNLOAD_BY_ID` – get failed download record (file name and write time)
  - `POND_UPDATE_FILENAMES_DOWNLOAD_STATUS` – mark file download status as SUCCESS/FAILED
- File mapping and progress
  - `POND_GET_SFTP_FILENAME_ID` – lookup file id by file path
  - `POND_INSERT_USAGE_TABLE_NAME_MAPPING` – insert file mapping (path, times, blank flag)
  - `POND_GET_RECORD_COUNT_FROM_DATABASE` – how many rows already processed for this file
  - `POND_UPDATE_USAGE_FILE_AS_BLANK` – mark known file as blank
- Core data movement
  - `POND_UPDATE_DEVICE_USAGE_FROM_STAGING` – merge from `POND_DEVICE_USAGE_STAGING` into canonical usage
  - `POND_DEVICE_SYNC` – update device-level data from canonical Pond usage

### Data tables referenced
- Staging: `DatabaseTableNames.POND_DEVICE_USAGE_STAGING`
- Retry/diagnostics: `DatabaseTableNames.POND_SFTP_FILE_DOWNLOAD_STATUS`
- Canonical usage and device tables: database-resident tables updated by `POND_UPDATE_DEVICE_USAGE_FROM_STAGING` and `POND_DEVICE_SYNC` (names defined centrally; updated via SPs).

### Operational knobs and safety
- `LimitNumberOfFilesPerInstance` – caps concurrent file ingestion per lambda instance
- `UsageRowsCountLimit` – caps rows per batch to enable incremental ingestion of large files
- `CheckFilesMissedThresholdDays` – SFTP listing lookback for missing files
- `DaysToKeep` – SFTP retention window for cleanup
- All DB calls are executed through `SqlQueryHelper` with retry policies from `PolicyFactory`.

### End-to-end sequence (condensed)
1) Trigger → enumerate providers → truncate staging (if hourly/daily)
2) For each provider → list SFTP → compare with already-downloaded → choose newest files
3) For each file → get file id + processed row count → download and parse → insert file mapping → stage rows via bulk copy → re-queue if partial
4) Enqueue `UpdatePondUsageFromStaging` → SP merges staging → enqueue `UpdateDeviceUsageFromPond` → SP updates device and usage tables
5) Cleanup SFTP files older than retention

### Key code touchpoints
- `AltaworxPondGetDeviceUsage.cs` – end-to-end orchestration, SQS, SFTP, SP invocations, bulk copy
- `PondUsageReportReader.cs` – record parsing and DataTable shaping (classic vs new CDR format)
- `SqlQueryHelper.cs` – SP execution helpers
- `ServiceProviderCommon.cs` – next-provider SP lookup
